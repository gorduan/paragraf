    **Root cause analysis for Gap 4 (Discovery API call fails):**

    The backend `DiscoverRequest` model has these fields:

    Where `LookupRequest` has `gesetz: str` and `paragraph: str` (e.g., "SGB IX" and "SS 152").

    The frontend `DiscoverRequest` interface has:
    ```typescript
    positive_paragraphs: { gesetz: string; paragraph: string }[];
    negative_paragraphs?: { gesetz: string; paragraph: string }[];
    ```

    The `handleExecuteDiscovery` function sends:

    **Possible issues to investigate and fix:**

    1. **The `fetchJson` function spreads `options` AFTER setting headers, which overwrites the Content-Type header.** Looking at line 289-292:
    ```typescript
    const res = await fetch(`${BASE_URL}${path}`, {
      headers: { "Content-Type": "application/json" },
      ...options,
    });
    ```
    If `options` does NOT include a `headers` field, the default `Content-Type` is preserved. The `discover` call passes `{ method: "POST", body: JSON.stringify(body) }` which does NOT include `headers`, so the default Content-Type IS preserved. This is NOT the bug.

    2. **The backend's `_resolve_point_id` method may fail to find the paragraph.** The backend code (api.py line 630-636):
    ```python
    resolved = await asyncio.gather(
        *[ctx.qdrant._resolve_point_id(p.paragraph, p.gesetz) for p in body.positive_paragraphs]

    for i, rid in enumerate(resolved):
        if rid is None:
            p = body.positive_paragraphs[i]
            raise HTTPException(404, f"Paragraph {p.paragraph} in {p.gesetz} nicht gefunden.")
    ```

    The `_resolve_point_id(paragraph, gesetz)` takes paragraph as FIRST argument, gesetz as SECOND. It searches for a point with matching `paragraph` and `gesetz` metadata fields in Qdrant. If the paragraph value from the frontend (e.g., "SS 152") doesn't EXACTLY match what's stored in Qdrant metadata, it won't find it.

    **The most likely issue:** The paragraph strings in `SearchResultItem` include the "SS" symbol (e.g., "SS 152 SGB IX" has `paragraph: "SS 152"`), but `_resolve_point_id` may expect just the number or a different format. Need to check what format Qdrant stores.

    ```typescript
    async function fetchJson<T>(
      path: string,
      options?: RequestInit
    ): Promise<T> {
      const res = await fetch(`${BASE_URL}${path}`, {
        headers: { "Content-Type": "application/json" },
        ...options,
      });
      if (!res.ok) {
        // Try to extract error detail from JSON response body
        let detail = `${res.status} ${res.statusText}`;
        try {
          const body = await res.json();
          if (body.detail) {
            detail = typeof body.detail === "string" ? body.detail : JSON.stringify(body.detail);
          }
        } catch {
          // Response body not JSON, use status text
        }
        throw new Error(detail);
      }
      return res.json();
    }
    ```

    **Root cause analysis for Gap 5 (UndoSnackbar not appearing):**

    The `UndoSnackbar` component uses `useEffect` with `onDismiss` in the dependency array:
    ```typescript
    useEffect(() => {
      const timer = setTimeout(onDismiss, duration);
      return () => clearTimeout(timer);
    }, [onDismiss, duration]);
    ```

    The `handleUndoDismiss` function in SearchPage.tsx is a plain function declaration:
    ```typescript
    function handleUndoDismiss() {
      setShowUndo(false);
      setUndoSnapshot(null);
    }
    ```

    Since this is not wrapped in `useCallback`, it is RECREATED on every render. This causes the `useEffect` to re-run on every render (cleanup old timer + set new timer). While this shouldn't prevent the snackbar from appearing, it could interact badly with React's batched state updates.

    The more likely issue is the SEQUENCE of state updates in `handleDiscoveryReset`:
    ```typescript
    function handleDiscoveryReset() {
      setUndoSnapshot({ examples: [...discoveryExamples], results: results });
      setDiscoveryExamples([]);
      if (previousResults) {
        setResults(previousResults);
        setPreviousResults(null);
      }
      setShowUndo(true);
    }
    ```

    In React 18+, state updates inside event handlers ARE batched. So `setShowUndo(true)` triggers a single re-render. The UndoSnackbar should render. However, `setDiscoveryExamples([])` clears the examples, which causes the DiscoveryExampleBar to re-render with 0 examples, which hides the "Zuruecksetzen" button. This is expected behavior.

    **Actual bug: The `showUndo` state IS set to true, but the UndoSnackbar renders INSIDE the `<div className="max-w-4xl mx-auto p-6">` container** (SearchPage line 446-452). Since the UndoSnackbar uses `fixed bottom-4 left-1/2` positioning, it should still appear regardless of parent container. However, if there's a parent element with `overflow: hidden` or `position: relative` with clipping, the fixed positioning could be broken.

    Wait — looking more carefully at the SearchPage render: the UndoSnackbar is rendered at line 446 as:
    ```tsx
    {showUndo && (
      <UndoSnackbar
        message="Beispiele entfernt"
        onUndo={handleUndo}
        onDismiss={handleUndoDismiss}
      />
    )}
    ```

    This is INSIDE the main `<div className="max-w-4xl mx-auto p-6">` container. Fixed positioning should still work from here.

    **The real bug might be simpler**: The `handleDiscoveryReset` function checks `if (previousResults)` before restoring. If `previousResults` is null (user hasn't executed discovery yet, just selected examples and clicked reset), then results stay the same, but `showUndo` is set to true. The snackbar SHOULD appear. Let me check if the undo snapshot is created correctly...

    Actually, I suspect the bug is that `handleUndoDismiss` as an unstable reference causes the `useEffect` to re-run immediately after mount, creating a rapid timer cycle. While the snackbar IS technically rendered for a frame, the immediate re-mount of the effect could cause visual flickering or the snackbar to appear for only milliseconds.

    **Fix for Gap 5:**

    1. **UndoSnackbar.tsx** — Remove `onDismiss` from the useEffect dependency array to prevent timer reset on every render. Use a ref to track the latest callback:
    ```typescript
    import { useEffect, useRef } from "react";

    export function UndoSnackbar({
      message,
      onUndo,
      onDismiss,
      duration = 5000,
    }: UndoSnackbarProps) {
      const onDismissRef = useRef(onDismiss);
      onDismissRef.current = onDismiss;

      useEffect(() => {
        const timer = setTimeout(() => onDismissRef.current(), duration);
        return () => clearTimeout(timer);
      }, [duration]);

      // ... rest unchanged
    }
    ```

    2. **SearchPage.tsx** — Wrap `handleUndoDismiss` and `handleUndo` in useCallback for stable references:
    ```typescript
    const handleUndo = useCallback(() => {
      if (undoSnapshot) {
        setDiscoveryExamples(undoSnapshot.examples);
        setResults(undoSnapshot.results);
        setUndoSnapshot(null);
      }
      setShowUndo(false);
    }, [undoSnapshot]);

    const handleUndoDismiss = useCallback(() => {
      setShowUndo(false);
      setUndoSnapshot(null);
    }, []);
    ```

    Note: `handleUndo` depends on `undoSnapshot` which is fine — the ref pattern in UndoSnackbar handles callback updates without resetting the timer.

    **Fix for Gap 6 (red chip text barely readable in dark mode):**

    In `DiscoveryExampleBar.tsx` line 53, the negative chip styling is:
    ```
    bg-error-50 dark:bg-error-900 text-error-700 dark:text-error-300 border-error-200 dark:border-error-700
    ```

    `text-error-300` on `bg-error-900` has low contrast. Change to `dark:text-error-200` for better readability:
    ```
    bg-error-50 dark:bg-error-900 text-error-700 dark:text-error-200 border-error-200 dark:border-error-700
    ```

    Also update the hover state from `dark:hover:bg-error-800` to keep contrast on hover.

    For symmetry, also check the positive chip: `dark:text-success-300` on `dark:bg-success-900` — change to `dark:text-success-200` for consistency.
